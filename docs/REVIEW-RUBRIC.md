# 运营后台 — 四镜头评审 Rubric(回测 + 审计 + 持续进化)

> 每次功能性改动后的质量门,**不只查技术正确,还要过三个体验/设计镜头**:
> ① 初次接触的运营者能否上手 ② 顶级 PM 视角的运营价值 ③ 顶级交互设计师的 UI/UX 打磨。
> 四镜头任一出现 P0/P1 都不算通过。本 rubric 是 `scripts/verify.sh`(自动可检项)
> 与「审计 panel」(并行 agent + 主线确认)共同遵循的判据。
> **每轮回测+审计后必做「持续进化循环」(见末节):反思 → 记台账 → 升级本 rubric/检查项。**

## 0. 机制总览

```
改动 → 回测(verify.sh:tsc + 路由 + needle + 体验回归)
     → 审计 panel(4 并行 reviewer:技术 / 运营者 / PM / 交互设计师,各按下方镜头)
     → 主线交叉验证(P0 必回源核实,不多数投票)
     → 修复循环 直到 四镜头 P0+P1 = 0
     → 持续进化:反思本轮被漏/被主人揪的点 → 记「经验教训台账」→ 升级 rubric/检查项
```

严重度:**P0** 阻断(技术崩坏 / 运营者完全无法上手 / PM 判断关键缺失或误导 / 版面崩坏失真)·**P1** 须修(明显障碍或价值/打磨缺口)·**P2** 建议 ·**P3** 备注。

## 镜头 A — 技术正确性(已有,保留)

由 `typescript-reviewer` / `code-reviewer` agent + `verify.sh` 覆盖。

- SSR/hydration:render 中不读未门控的 persist / `Date.now()`;SVG/宽度数值四舍五入。
- Next 16:`params`/`searchParams` await;server/client 边界正确。
- 状态机正确、Zustand 不可变、effect 依赖、async confirm 流。
- a11y 基线:可点元素可键盘达、aria、对话框焦点。
- 类型安全、无不安全 cast、无 secret/XSS。

## 镜头 B — 初次运营者上手(First-Touch Operator)

**核心问题:一个第一次打开这个后台、不看文档的运营者,能不能在 30 秒内知道"这是什么 / 我能做什么 / 现在有什么要处理",并顺利完成一次操作?**

逐页检查清单:
- [ ] **落地即态势**:打开页面立刻能感知"现在健不健康 / 有什么待办",而非只见菜单或标题。(首页尤其:应是指挥台,不是目录)
- [ ] **信息气味**:列表行 / 卡片 / 菜单项带**状态色 + 数字 + 趋势**,一眼看出"哪条有事",不是纯文本。
- [ ] **操作可发现**:主 CTA 视觉明确;按钮禁用时**内联说明为什么**(如「操作理由必填」)。
- [ ] **指标可理解**:关键指标(覆盖率 / 留存 / CVR / 风险分)有口径 tooltip 或说明,不假设运营者懂黑话。
- [ ] **空态有引导**:未建 / 无数据时说明"这个模块控制什么 / 典型操作 / 风险点",不是空白或纯 TODO。
- [ ] **定位清晰**:面包屑 + active 高亮让运营者随时知道"我在哪 / 能去哪";域分类清晰。
- [ ] **节奏可知**:当前 12 月运营阶段(P1–P6)及其重心在界面可感知。
- [ ] **防迷路**:危险/不可逆操作有二次确认;跨域跳转有路径。

判定:
- 落地完全无态势 / 主操作找不到 / 指标无从理解 → **P0**。
- 信息气味弱 / 空态空洞 / 节奏不可知 → **P1**。

## 镜头 C — 顶级 PM 运营价值(Operational Value)

**核心问题:运营者用这个页面能不能又快又稳地达成运营目标(拉新/转化/兑付安全/合规)?信息颗粒度够不够支撑决策?**

逐页检查清单:
- [ ] **决策就绪**:页面回答"该不该做、做了什么后果"——放大资金流出的操作显示**兑付覆盖率核验/红线**;改参数显示**联动**(派发 D5/F3/G5、触发 J1 等)。
- [ ] **信息颗粒度**:数字带**环比 / 趋势 / 基线对比 / SLA**,不是孤立值;有 sparkline / delta / 分布。
- [ ] **全局态势可达**:覆盖率 / 资金水位 / 告警 / 节奏在该页可见或一键可达。
- [ ] **效率杠杆**:筛选 / 已存视图 / 批量 / 键盘 / 跨域深链预填,减少点击与等待。
- [ ] **防错可信**:操作确认、理由校验、审计留痕在敏感操作处可见。
- [ ] **业务意图对齐**:每个杠杆服务 §17.2 八项 KPI / 12 月节奏 / 转化优先 / 兑付安全,无"为做而做"。
- [ ] **角色相关**:该角色看到的是其战场所需(RBAC 过滤合理),不被无关信息淹没。

判定:
- 放大流出无红线核验 / 关键决策信息缺失或误导 / 无审计 → **P0**。
- 颗粒度不足(只有孤立数字)/ 无效率杠杆 / 态势不可达 → **P1**。

## 镜头 D — 顶级交互设计师(UI/UX 打磨)

**核心问题:这界面像不像被精心打磨过的成熟产品?有没有"对不齐 / 大段留白 / 关键控件不标数值 / 凭截图臆测交互"这类低级失真?**(2026-06-02 主人新增镜头)

逐页/逐组件检查清单:
- [ ] **无参差留白**:同一行卡片**等高、底边齐平**;卡片 flex 填满网格,footer/CTA `mt-auto` 底对齐,不留大段空白(尤其 span-2 配 1-col)。
- [ ] **对齐一致**:网格列 / 卡片内边距 / 圆角 / 间距遵循统一节奏;同类元素左右边、基线对齐。
- [ ] **控件自带数据**:进度条 / 仪表 / 分区条**在标记处直接显示当前数值**(如覆盖率条标记上写 89.8%);阈值 / 量程刻度齐全(红线 / 健康线带 %),不让人对刻度估。
- [ ] **视觉层级**:一屏有明确主角(最大最亮),余者退为配角;字重 ≤600、字号档位克制;亮底文字用 on-brand 深色。
- [ ] **状态反馈**:hover / active / focus 真实生效(警惕内联 `style` border 覆盖 Tailwind `hover:` 致失效);可点元素 rest 态即有可点气味。
- [ ] **形态忠实**:复刻设计稿交互形态(抽屉 / 弹窗 / 模态 / 动画)前**读设计稿源码(jsx+css)确认**,不靠截图臆测(踩过:侧滑抽屉做成下拉)。
- [ ] **完整不缺**:设计稿元素**不静默省略**;省略必显式告知主人 + 给理由;数据缺口用诚实标注的 mock 补,不跳过。
- [ ] **改动整体一致**:按设计稿重构先列全元素清单,**一次覆盖到位**(漏斗 / 节奏 / KPI / 顶栏…),不留半套旧样式让主人逐个揪。

判定:
- 大段留白 / 严重错位 / 控件该有数值却没有,使界面显廉价或误读 → **P0**。
- 局部不齐 / hover 失效 / 形态与设计稿不符 / 元素被静默省略 → **P1**。

## 镜头 E — 可操作性(Operability)(2026-06-03 主人催生:"看着能点实则点了没反应")

**核心问题:每个"看着能交互"的控件是否真的能操作?每个前端杠杆是否真有可改的后台入口?**——这是本项目反复踩的坑:好看 ≠ 可操作。

逐页/逐控件检查清单:
- [ ] **无死控件**:每个 `<Btn>`/`<button>`/`<Toggle>`/可点元素都有 `onClick`/`href`,点击产生**可观测效果**(弹 modal/drawer / toast / 导航 / 状态翻转 / 真实下载)。源码红线:`<Toggle …/>` 与 `<Btn …>` **不得无 onClick**(踩过:G1 staking kill 开关、脱敏/账本导出、新建规则)。
- [ ] **每个配置杠杆都可调**:每个配置型参数(价/率/APY/cap/门槛/版税/倍率/供给/残值/冷却/开关)都有**自己的「调整」入口**;不是"每 Tab ≥1 按钮"就算过(踩过:G4 Genesis 单价/分红率只读、1 个 pause 按钮蒙混)。
- [ ] **调整弹窗可编辑**:"调整/配置"类 操作确认弹窗必须含**可编辑「目标新值」控件**(开关/选项/数字/文本)+ 当前→新值预览 + 未设值禁用应用;不能只是确认抽象动作(踩过:操作确认弹窗纯确认框,窗口内无可操作项)。
- [ ] **弹窗/抽屉不是死胡同**:打开后有真实输入/操作(表单 + 提交、行操作),不是只有取消/关闭。
- [ ] **纯动作弹窗** (放行/退款/pause/冻结/封禁/确认拦截) 仅确认 + 操作理由即可,**无需**目标值控件(对它们加编辑控件反而错)。
- [ ] **导航不冗余**:侧栏手风琴已导航子模块时,**页内不再放横向 Tab**(路由直出该子模块内容)。
- [ ] **前端杠杆 grounded**:逐杠杆核对以 **H5 真 app 路由**为准(见 `docs/FRONTEND-LEVER-MAP.md`),不对 PRD 臆测。

判定:死控件 / 配置杠杆无调整入口 / 调整弹窗无编辑控件 → **P0**;弹窗死胡同 / 冗余页内 Tab → **P1**。

### 可操作性审计方法(机器可检 harness)
1. **配置覆盖**(找只读杠杆):`grep '<KV k=' 各视图 | 配置关键词 | 排除 Btn|调整|前往|Badge|权威|server` → 只读配置参数 = 缺调整入口。
2. **死控件 — 源码**:`grep -E '<Toggle|<Btn' 各视图 | grep -v onClick`(排除 type=submit)→ 无 onClick 的控件。
3. **死控件 — 运行时**:`node _overview-variants/audit-operability.mjs`(playwright)逐路由点击每个内容按钮,**effect = overlay(.modal/.drawer/scrim/[role=dialog]) 增加 OR toast(内联 `div[style*="position: fixed"]`)增加 OR url 变化**;无 effect=疑似死。
4. **误报铁律(必确认源码再下结论)**:① toast 是**单例不堆叠**→快速连点时 toast 计数不增 = 假阴性;② 文件下载(Blob+a.click)**无 DOM 变化** = 测不到。故运行时报"死"的,一律回源 `grep onClick` 确认是否其实接了 setToast/下载,再决定。
5. **正则全集**:控制/affordance 检测动词正则要全(新增|新建|发布|发送|设为|停止|调整|调价|确认|冻结…);图标型 affordance 按结构(`a[href]>svg`)断言,不靠字形字符。

## 自动可检 vs 需 panel 评

| 维度 | verify.sh 体验回归段(自动 tripwire) | 审计 panel(agent + 主线) |
|---|---|---|
| 技术 | tsc / 路由 200 / needle / 路由计数 | typescript/code reviewer |
| 运营者 | 页面非空、状态信号在位、脚手架有引导文案 | 镜头 B 全清单 |
| PM | 旗舰页有颗粒度信号(环比/积压/风险)、覆盖率/节奏 needle 在位 | 镜头 C 全清单 |
| 交互设计师 | 关键填充/洞察 needle(如「最大流失」)、控件数值 needle 不退化 | 镜头 D 全清单(留白/对齐/形态忠实需肉眼+源码核) |
| 可操作性 | 死控件 grep(`<Toggle`/`<Btn` 无 onClick)+ 配置覆盖 grep + `audit-operability.mjs` 运行时点测 | 镜头 E 全清单(调整弹窗可编辑/弹窗非死胡同/每杠杆可调 — 误报需回源核实) |

> verify.sh 只能守"信号没消失"这类可 grep 的退化;**质量是否真到位由 panel 四镜头评 + 主线交叉验证**。

## 审计 panel 执行(每次功能改动后)

0. **机器可检先行**(改动含控件/配置时必跑,先于起 agent):① 配置覆盖 grep ② 死控件 grep ③ `node _overview-variants/audit-operability.mjs` 运行时点击扫描(误报必回源核实,见镜头 E 方法)。先把可机检的死控件/缺调整入口清掉,再上 panel 评定性维度。
1. 并行起 5 个 read-only reviewer:
   - **技术**(typescript-reviewer,scope=改动文件)
   - **初次运营者**(general-purpose,带本 rubric 镜头 B,review 渲染体验/JSX)
   - **顶级 PM**(general-purpose,带本 rubric 镜头 C,对齐 §17.2 KPI/phase/兑付)
   - **顶级交互设计师**(general-purpose,带本 rubric 镜头 D,核留白/对齐/控件数值/形态忠实;UI 改动必起)
   - **可操作性**(general-purpose,带本 rubric 镜头 E,核死控件/每配置杠杆可调/调整弹窗可编辑/弹窗非死胡同;含控件或配置的改动必起)
2. 主线汇总 findings,**对每条 P0/P1 回源核实(读代码/跑 preview/读设计稿源码)再裁定,不多数投票**。
3. 按严重度修复 → 重跑 verify.sh + 受影响镜头复审 → 循环到五镜头 P0+P1=0。
4. 纯布局/文案微调(无逻辑)可只跑 verify.sh + 主线自检镜头清单,免起 agent;但**涉及视觉/交互形态的改动必过镜头 D,涉及控件/配置的改动必过镜头 E(+机器扫描)**。
5. **持续进化(必做收尾)**:本轮结束后执行末节循环 —— 反思 → 记台账 → 升级 rubric/needle/memory。

参考:本机制扩展自项目 H5 的 `nexion-audit`(3-agent 模型),此处把"运营者 / PM / 交互设计师"三镜头固化为常驻维度。

## 持续进化循环(每轮回测+审计后必做)

> 主人指令(2026-06-02):**每次回测和审计后吸取教训、反思总结,再优化进化流程**。

1. **反思**:本轮哪些问题是被 verify/audit 漏掉、或被主人当面揪出来的?
2. **归因**:根因是什么(缺某镜头 / 没读设计稿源码 / 静默省略 / 半套改动 / 凭记忆臆测 …)?
3. **记台账**:在下方「经验教训台账」追加一行(日期 · 现象 · 根因 · 固化的新规则)。
4. **升级**:把可推广的教训固化为 —— 本 rubric 的新 checklist 项 / verify.sh 的新哨兵 needle / memory 规则。让同类问题**下次被机制挡住**,而非靠主人再发现。
